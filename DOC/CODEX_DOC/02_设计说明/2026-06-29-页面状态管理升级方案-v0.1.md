# 页面状态管理升级方案 v0.1

## 1. 背景

当前基准时间轴页面已经具备可用的投影数据集切换、双轴显示、缩放、横向拖动、故事线展开折叠和 Inspector 选中解释能力。但这些能力目前只是散落在运行变量和 DOM 输入框中，没有形成稳定的“状态系统”。

这导致三个直接问题：

- 页面刷新后，缩放比例、横向停留位置、轴模式、选中对象和故事线展开状态无法稳定恢复。
- 多个投影数据集之间共用一套临时视图变量，无法记住“这个数据集上次展开了哪些故事线、停在哪个位置”。
- `track.expanded` 直接写在投影数据对象上，视图状态和数据事实混在一起，后续会干扰数据模型产品化。

## 2. 目标

本轮升级目标是建立第一版页面状态管理底座，使页面具备可解释、可保存、可恢复、可重置的状态行为。

必须支持：

- 页面级状态：当前数据集、轴模式、时间范围、缩放比例、横向位置、选中对象。
- 数据集级状态：每个投影数据集各自的故事线展开状态、缩放比例、横向位置、时间范围、选中对象。
- 本地持久化：刷新页面后恢复最近使用状态。
- 显式重置：提供用户可见的“重置状态”按钮。
- 状态隔离：不同投影数据集之间的视图状态不能互相污染。

## 3. 非目标

本阶段不做：

- 跨设备同步。
- 用户账号体系。
- 状态文件导入导出。
- 多人协作状态。
- 正式对象数据库状态。
- 将页面拆成完整前端框架工程。

这些能力可以在后续产品化阶段扩展，但不进入第一版状态管理底座。

## 4. 当前状态来源

当前页面的状态分布如下：

| 状态 | 当前位置 | 问题 |
| --- | --- | --- |
| 缩放比例 | `state.scale` | 只在内存中，刷新丢失。 |
| 横向位置 | `state.offsetX` | 只在内存中，刷新丢失。 |
| 轴模式 | `state.axisMode` | 只在内存中，刷新丢失。 |
| 当前选中对象 | `state.selected` | 只在内存中，切数据集和刷新时没有统一恢复规则。 |
| 详情模式 | `state.detailMode` | 属于临时交互态，不应默认持久化。 |
| 当前数据集 | `activeStoryId` | 只在内存中，刷新依赖 manifest 默认值。 |
| 故事线展开状态 | `track.expanded` | 写在数据对象上，混淆数据事实和视图状态。 |
| 时间范围 | `timeline.start/end` 和 DOM input | 没有统一保存恢复入口。 |

## 5. 状态模型

第一版采用两层状态：

```json
{
  "version": 1,
  "activeStoryId": "story-current",
  "datasets": {
    "story-current": {
      "axisMode": "realTime",
      "timelineStart": "2025-07-05T09:00",
      "timelineEnd": "2025-07-10T02:00",
      "scale": 0.72,
      "offsetX": 0,
      "selected": {
        "type": "event",
        "id": "E005"
      },
      "expandedTrackIds": ["interview", "temple", "zhang"]
    }
  }
}
```

### 5.1 页面级状态

页面级状态只保存：

- `version`
- `activeStoryId`
- `datasets`

它负责说明当前打开的是哪个数据集，以及每个数据集对应的视图状态。

### 5.2 数据集级状态

数据集级状态保存：

- `axisMode`
- `timelineStart`
- `timelineEnd`
- `scale`
- `offsetX`
- `selected`
- `expandedTrackIds`

这些状态必须按 `activeStoryId` 隔离。同一个故事线在不同数据集中的展开状态不能共用。

### 5.3 不持久化状态

以下状态不持久化：

- `hover`
- `isDragging`
- `dragStartX`
- `dragStartOffset`
- `moved`
- `detailMode`
- `#event-detail`、`#chapter-detail` 等调试/演示 hash

原因：它们属于瞬时交互状态，刷新后恢复会制造误解。

## 6. 持久化策略

第一版使用浏览器 `localStorage`。

建议 key：

```text
hs-novel-editor:timeline-view-state:v1
```

保存时机：

- 切换投影数据集前，先保存当前数据集状态。
- 切换投影数据集后，恢复目标数据集状态。
- 缩放变化后保存。
- 横向拖动结束后保存。
- 轴模式变化后保存。
- 时间范围应用成功后保存。
- 故事线展开/折叠后保存。
- 选中对象变化后保存。
- 点击“全展”后保存。
- 点击“重置状态”后删除当前数据集状态并立即恢复默认状态。

保存节流：

- 拖动过程中不连续写入。
- `pointerup` 后保存一次。
- 缩放滑杆输入可以按最后一次值保存，必要时加轻量 debounce。

## 7. 恢复策略

页面启动顺序建议改为：

1. 读取投影数据集 manifest。
2. 读取本地状态对象。
3. 确定 `activeStoryId`：
   - URL 或用户显式选择优先。
   - 本地状态中的 `activeStoryId` 次之。
   - manifest 默认值兜底。
4. 加载目标数据集。
5. 根据该数据集的 `DatasetViewState` 恢复：
   - 时间范围。
   - 轴模式。
   - 展开的故事线。
   - 缩放比例。
   - 横向位置。
   - 选中对象。
6. 如果某个状态引用的数据对象不存在，丢弃该局部状态并继续启动。

## 8. 交互按钮调整

当前右上工具组包含：

```text
缩小 / 复位 / 放大 / 全展 / 收起
```

建议调整为：

```text
缩小 / 复位 / 放大 / 全展 / 重置状态
```

### 8.1 保留“复位”

“复位”保留为视口复位，只影响：

- `scale`
- `offsetX`
- `detailMode`

它不清除当前数据集保存状态，只是把当前视图拉回默认观察位置。

### 8.2 移除“收起”

“收起”价值较低，原因：

- 它只是批量折叠故事线。
- 用户真正需要的是恢复一个可理解的默认状态。
- 有了按数据集保存的展开状态后，“收起全部”会变成破坏用户状态的危险操作。

### 8.3 新增“重置状态”

“重置状态”替代原“收起”位置。

点击后行为：

1. 删除当前数据集的 `DatasetViewState`。
2. 恢复该数据集的默认展开状态。
3. 恢复默认轴模式、时间范围、缩放和横向位置。
4. 清空选中对象。
5. 隐藏事件检查器和章节详情。
6. 更新 Inspector 到未选中状态。
7. 写入新的本地状态。

按钮文案建议：

```text
重置状态
```

如果按钮宽度受限，可用：

```text
重置
```

但 Tooltip 或 Inspector 状态提示应说明它是“重置当前数据集状态”，不是只重置缩放。

## 9. 实现拆分

建议拆成四个小步骤实现，避免一次改动过大。

### Step 1：状态读写层

新增纯函数：

- `loadPersistedState()`
- `savePersistedState(nextState)`
- `getDatasetViewState(storyId)`
- `setDatasetViewState(storyId, viewState)`
- `clearDatasetViewState(storyId)`

要求：

- localStorage 读取失败时不影响页面启动。
- JSON 解析失败时丢弃旧状态并继续。
- `version` 不匹配时丢弃旧状态。

### Step 2：状态采集与应用

新增：

- `captureCurrentDatasetState()`
- `applyDatasetViewState(viewState)`
- `applyDefaultDatasetViewState()`

并把 `track.expanded` 的读写逐步收口到状态应用函数中。

### Step 3：关键事件接入保存

接入以下入口：

- `setScale`
- `setAxisMode`
- `applyTimelineRange`
- `selectRegion`
- 故事线展开/折叠逻辑
- `expandAll`
- 数据集切换前后
- canvas 拖动结束

### Step 4：交互按钮改造

- 将 `collapseAll` 按钮替换为 `resetDatasetState`。
- 保留 `expandAll`。
- 保留 `resetView` 作为视口复位。
- 为重置状态增加确认策略。

## 10. 重置确认策略

第一版可不弹系统确认框，但需要提供明确状态反馈：

```text
当前数据集状态已重置
```

如果后续用户认为误触成本高，再增加轻量确认。

## 11. 风险与边界

| 风险 | 处理 |
| --- | --- |
| 数据集结构变更导致旧状态引用失效 | 恢复时检查对象是否存在，不存在则丢弃该字段。 |
| localStorage 被禁用或写满 | 捕获异常，降级为内存状态，不阻塞页面。 |
| 旧 hash 演示路由干扰状态恢复 | 调试 hash 只允许选中对象，不允许自动打开详情面板。 |
| `track.expanded` 仍写在数据对象上 | 第一版允许内部应用时写入，但事实源改为 `expandedTrackIds`。后续拆分运行层时再彻底移除数据对象变异。 |
| “复位”和“重置状态”语义混淆 | UI 文案、按钮位置和状态栏反馈必须区分。 |

## 12. 验收标准

### 12.1 机器校验

需要新增或更新校验：

- 状态 key 名称和 version 存在。
- 初始路由不自动展开详情面板。
- `collapseAll` 不再作为主工具按钮存在。
- `resetDatasetState` 入口存在。

### 12.2 浏览器验收

至少验证：

1. 展开一条故事线，刷新页面后仍保持展开。
2. 缩放并横向拖动，刷新后恢复位置。
3. 切换到另一个数据集，状态独立保存。
4. 切回原数据集，恢复原数据集的展开和视图位置。
5. 点击“重置状态”，当前数据集回到默认状态。
6. 点击“复位”，只重置视口，不清除故事线展开状态。
7. 带 `#event-detail` 刷新不会自动显示事件检查器。

## 13. 推荐结论

该功能需要做底层改造，但不需要重写页面。

推荐先实现本地状态底座，以 `localStorage + versioned state schema + dataset state map` 为第一阶段目标。这样能立刻解决刷新丢状态、多数据集状态互相污染、展开状态写在数据对象上的问题，同时为后续“工作区状态文件”和正式产品化状态模型留下清晰接口。
