# 工程化改造与前端解耦设计 v0.1

## 1. 背景

当前基准时间轴页面已经从一次性验证工具演进为有持续价值的叙事结构管理工具。页面已经具备投影数据集切换、真实时间轴与章节等宽轴、Canvas 绘制、Inspector、状态持久化、宽屏模式和二维拖拽等能力。

但当前运行实现仍位于：

```text
原型包/2026-06-22-叙事验证工具-基准时间轴原型-v1/source/index.html
```

该文件同时承担 HTML 模板、CSS、数据加载、状态管理、轴投影、几何计算、Canvas 绘制、命中测试、Inspector 更新和 DOM 事件绑定。文件规模已经超过 2000 行，继续在该文件内累加功能会带来三个问题：

- 每次 UI 调整都可能影响状态、绘制和命中测试。
- 数据事实、视图状态和渲染临时值边界不清。
- 后续章节函数、报告、同步建议等功能没有稳定扩展位置。

因此，本轮需要建立工程化改造方案，将当前原型页面迁移为可维护的小型前端工程。

## 2. 目标

本设计目标是定义第一版前端工程化迁移边界，使后续实现能从原型单文件转入正式工程根。

必须达成：

- 建立新的正式运行工程根 `app/`。
- 保留当前原型包作为行为基线、视觉基线和历史事实源。
- 采用渐进迁移，不做大爆炸重写。
- 将数据读取、状态管理、时间轴模型、Canvas 渲染、命中测试、Inspector 和启动流程拆成清晰模块。
- 保持现有用户可见行为，包括宽屏模式、二维拖拽、双轴模式、数据集状态持久化和 Inspector 行为。
- 让现有静态校验脚本可以迁移到新工程，并继续作为回归证据。

## 3. 非目标

本阶段不做：

- 不重做 UI 视觉风格。
- 不切换到 React、Vue 或其他组件框架。
- 不引入后端服务。
- 不建立正式小说对象数据库。
- 不移动或提交 `验证作品/`。
- 不直接修改外层小说正文、设定和成熟支撑材料。
- 不把历史原型包整体搬进正式文档目录。

## 5. 工程根决策

正式运行工程根建议采用：

```text
app/
```

理由：

- 语义直接，表示当前项目的正式可运行应用。
- 与 `原型包/` 明确隔离，避免继续把产品实现写入历史原型目录。
- 后续可自然承载 `package.json`、`src/`、`public/`、`scripts/` 和测试入口。
- 不干扰已有 `DOC/CODEX_DOC/` 工程文档根。

`原型包/` 的新定位：

- 保留为历史原型资产根。
- 当前 V1 原型在迁移期作为行为基线。
- 新功能迁入 `app/` 后，原则上不再继续向原型包追加产品功能。
- 若必须修正原型包，只允许做基线修复或对照验证修复，并在提交中说明原因。

## 6. 技术路线

第一版采用 Vite + 原生 JavaScript ES Modules，不引入前端框架。

理由：

- 当前核心是 Canvas 命令式绘制，框架组件化收益有限。
- 原型代码可以以较低风险迁移成模块。
- Vite 已经被当前本地启动流程使用，迁移成本低。
- 先拆边界，再谈框架；避免把结构性债务包进新框架。

后续如果需要复杂面板、表单、路由和多页面协作，再评估 React 或其他框架。

## 8. 模块边界

### 8.1 `data/`

职责：

- 加载投影数据集清单。
- 加载具体投影数据集。
- 做轻量结构校验和运行时归一化。
- 暴露只读数据对象给页面层。

禁止：

- 不写入视图状态。
- 不保存 localStorage。
- 不把故事线展开状态写回投影数据事实。

### 8.2 `state/`

职责：

- 管理页面级状态。
- 管理每个数据集独立的视图状态。
- 读写 `localStorage`。
- 处理状态版本、异常降级和失效对象引用。

边界：

- 状态层只描述“当前看到了什么、停在哪里、选中了什么”。
- 不承担 Canvas 绘制。
- 不直接操作 DOM。

### 8.3 `timeline/`

职责：

- 建立真实时间轴和章节等宽轴投影。
- 计算行布局、坐标、可视范围和裁剪。
- 生成绘制模型。
- 执行 Canvas 绘制。
- 执行命中测试。
- 处理拖拽、缩放等时间轴交互意图。

拆分原则：

- `axis-projection.js` 和 `geometry.js` 应尽量保持纯函数。
- `renderer.js` 只消费渲染模型和 Canvas context。
- `hit-test.js` 不更新选择状态，只返回命中的区域。
- `interactions.js` 把指针事件转成 viewport state 变更，不直接写业务数据。

### 8.4 `inspector/`

职责：

- 根据当前选择对象生成 Inspector 展示模型。
- 渲染 Inspector 摘要、事件说明和详情入口状态。
- 维护事件检查器和章节函数入口的按钮可用性。

边界：

- Inspector 解释对象，不修改投影数据。
- Inspector 可以触发进入详情或章节函数，但不直接承担路由解析。

### 8.5 `app/`

职责：

- 启动应用。
- 获取 DOM 引用。
- 绑定事件。
- 调度数据、状态、时间轴和 Inspector。
- 处理 URL/hash 的初始化行为。

边界：

- `bootstrap.js` 可以有副作用。
- 其他模块应尽量让副作用显式。

## 9. 状态迁移要求

迁移后必须保留当前状态能力：

- 当前数据集。
- 每个数据集独立的轴模式。
- 每个数据集独立的时间范围。
- 每个数据集独立的缩放比例。
- 每个数据集独立的横向位置和纵向位置。
- 每个数据集独立的宽屏模式。
- 每个数据集独立的选中对象。
- 每个数据集独立的故事线展开状态。

第一版可以继续沿用当前 key：

```text
hs-novel-editor:timeline-view-state:v1
```

但需要把读写集中在 `state/persisted-state.js`，页面其他模块不得直接访问 `localStorage`。

## 10. 数据迁移要求

第一版 `app/public/data/` 可以复制当前 `source/data/` 内容，作为稳定运行样本。

要求：

- 不引用 `验证作品/`。
- 不把 `验证作品/` 纳入 app 数据目录。
- 不把投影数据集误称为正式小说数据库。
- 数据文件路径变化后，启动文档和校验脚本必须同步更新。

后续如果需要让 app 读取外部素材，只能通过明确的导入流程或只读配置实现，不能直接把本地素材区变成 Git 管理内容。

## 11. 迁移阶段

### P2.4.1 建立 app 运行壳

产物：

- `app/package.json`
- `app/index.html`
- `app/src/main.js`
- `app/styles/`
- `app/public/data/`

验收：

- `npm install` 或等价安装流程可完成。
- `npm run dev` 可启动页面。
- 页面加载默认投影数据集。
- 视觉上与当前原型主页面保持结构一致。

### P2.4.2 抽出数据与状态层

产物：

- `src/data/*`
- `src/state/*`
- 状态相关校验脚本迁移。

验收：

- 刷新后恢复数据集、轴模式、缩放、横向/纵向位置、宽屏模式和选中对象。
- 切换数据集时状态互不污染。
- 重置状态只清除当前数据集状态。

### P2.4.3 抽出时间轴模型与 Canvas 层

产物：

- `src/timeline/axis-projection.js`
- `src/timeline/geometry.js`
- `src/timeline/row-layout.js`
- `src/timeline/renderer.js`
- `src/timeline/hit-test.js`
- `src/timeline/interactions.js`

验收：

- 真实时间轴和章节等宽轴行为保持一致。
- 事件块最小宽度、命中优先级和标签裁剪规则保持一致。
- 横向和纵向拖拽保持可用。
- 宽屏模式不隐藏 Inspector。

### P2.4.4 抽出 Inspector 与交互调度

产物：

- `src/inspector/*`
- `src/app/event-bindings.js`
- `src/app/route.js`

验收：

- 选中章节、故事线、人物和事件时 Inspector 摘要一致。
- 刷新带 `#event-detail` 不自动展示事件检查器。
- 事件检查器只在用户明确进入时显示。
- 章节函数按钮状态与当前原型一致。

### P2.4.5 切换正式入口并冻结原型

产物：

- 更新 `CODEX_START_HERE.md`。
- 更新 `DOC/CODEX_DOC/README.md`。
- 更新 `DOC/CODEX_DOC/02_设计说明/00-设计事实源索引.md`。
- 更新验收入口和机测记录。

验收：

- 新会话默认从 `app/` 启动。
- 当前原型包只作为历史事实源和视觉回归基线。
- 所有迁移前关键校验在 `app/` 下有等价入口。

## 12. 验收方法

迁移实现期间，每个阶段至少执行：

```powershell
npm run check
npm run dev
```

并保留以下行为验收：

1. 默认页面能加载当前投影数据集。
2. 真实时间轴与章节等宽轴切换正常。
3. 缩放、复位、放大、全展、重置状态、宽屏按钮正常。
4. Canvas 可横向和纵向拖拽。
5. Inspector 在无选中、章节、故事线、人物、事件状态下展示一致。
6. 刷新后恢复当前数据集状态。
7. 切换数据集后状态隔离。
8. `验证作品/` 不进入 Git。

页面迁移阶段必须使用真实浏览器验证，不得只用静态脚本替代。

## 13. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 大量代码迁移导致行为漂移 | 以当前原型为行为基线，分阶段迁移，每阶段保留脚本和浏览器验证。 |
| 过早引入框架扩大改造面 | 第一版使用原生 ES Modules，只拆边界，不换技术范式。 |
| 数据事实和视图状态继续混杂 | 明确 `data/` 只读、`state/` 管视图状态，禁止页面散落写 localStorage。 |
| 新旧入口并存导致混乱 | P2.4.5 前明确“原型仍是运行基线”；切换后明确“app 是正式入口”。 |
| 文档和启动命令不同步 | 每个迁移阶段完成时同步 `CODEX_START_HERE.md`、事实源索引和验收入口。 |
| 用户误以为工程化会改变 UI | 本阶段不重做 UI；任何视觉变更必须单独提出并确认。 |

## 14. 推荐结论

本项目应立即停止把产品功能继续堆入原型单文件，并进入 `P2.4 工程化改造与前端解耦`。

推荐执行路径：

1. 先建立 `app/` 运行壳。
2. 再迁移数据层和状态层。
3. 然后迁移时间轴几何、渲染和命中测试。
4. 最后迁移 Inspector 和启动调度。
5. 验证通过后切换正式入口，冻结原型包。

该方案比直接重写保守，也比继续维护单文件更可控。它把当前已经验证有效的页面行为保留下来，同时为后续章节函数、报告、同步建议和更正式的数据模型留下工程空间。
