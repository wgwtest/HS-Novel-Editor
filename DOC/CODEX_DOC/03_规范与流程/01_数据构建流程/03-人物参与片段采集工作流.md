# 人物参与片段采集工作流

## 1. 目的

人物参与片段用于回答：某个人在某个故事事件中具体承担了什么作用。

它不是人物传记，也不是事件标题的复制。它服务于基准时间轴中的人物子线条和 Inspector 聚合说明。

## 2. 输入前提

采集人物参与片段前，应先有已确认或待确认的故事事件。

人物参与片段必须依附于事件。没有明确事件时，不应单独生成子线条。

## 3. 采集问题

对每个可能参与人物，依次判断：

1. 这个人物是否在正文或明确规划中参与了该事件。
2. 他/她做了什么。
3. 是主动推动、被动承接、误读、压责、保护、转译、诱导，还是旁观校正。
4. 这个动作是否改变了关系、判断或后续条件。
5. 参与是正文可见，还是后台状态。

如果只能写出“参与”“出现”“相关”，说明不应生成独立人物参与片段。

## 4. `roleBrief` 规则

`roleBrief` 是人物子线条的画布显示短语。

规则：

- 推荐 6-12 字。
- 允许范围 5-15 字。
- 使用“动作 + 对象/方向”结构。
- 不复制故事事件标题。
- 不使用“参与”“相关”“出现”等空词。
- 不把故事线名当短述。

示例：

| 可接受 | 不可接受 |
| --- | --- |
| `承接现场求助` | `参与事件` |
| `误读授权` | `C5 电视台线` |
| `建立材料底盘` | `正式任务授权一周跟进` |
| `截断羞辱` | `出现` |

## 5. 字段输出

当前投影数据集中的人物参与片段至少应能生成：

```yaml
characterParticipation:
  kind: character
  track: ''
  character: ''
  participantName: ''
  title: ''
  roleBrief: ''
  action: ''
  relationChange: ''
  inspectorSummary: ''
  visibility: explicit
```

其中：

- `title` 应等于 `roleBrief`。
- `action` 写清人物在事件中的具体动作。
- `relationChange` 写清关系或判断变化。
- `inspectorSummary` 给 Inspector 使用，不应是模板空话。
- `visibility` 区分 `explicit` 和 `background`。

## 6. 后台状态处理

人物持续思考、整理材料或受影响，如果正文没有呈现，不应拉长故事事件。

正确处理方式：

- 如果只是后台持续：标记为人物线后台段或采集备注。
- 如果后来进入正文：在进入正文的位置新建事件或事件投影。
- 如果改变了人物行动输出：再判断是否形成事件或人物参与片段。

## 7. 校验入口

人物参与片段进入投影 JSON 后，应至少通过：

```powershell
node scripts/validate-character-participation.mjs
node scripts/validate-storyline-preview-quality.mjs
```

如果校验失败，应优先回到源定义或采集表修正，不优先手改运行投影 JSON。
