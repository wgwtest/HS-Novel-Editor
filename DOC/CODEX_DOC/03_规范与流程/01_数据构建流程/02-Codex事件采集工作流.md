# Codex 事件采集工作流

## 1. 目的

本文定义 Codex 如何从小说素材中采集候选事件，并把确认后的结果交给源定义或后续生成流程。

事件采集的目标不是把每个场景、段落或动作都写成事件，而是识别能改变叙事状态的变化单元。

## 2. 前置读取

采集前按最小必要上下文读取：

1. 当前任务说明。
2. 目标章节正文或章节梗概。
3. 相邻章节摘要，必要时读取全文。
4. `设计说明/2026-06-24-事件拆分规则与智能体执行准则-v1.0.md`。
5. 相关故事线定义或事件脉络。

不默认全量读取整部小说。需要全书级判断时，应先说明为什么需要扩大范围。

## 3. 工作步骤

### Step 1：列候选变化

先列出候选变化，不直接写 JSON：

- 哪些人物判断发生变化。
- 哪些关系、权力、信息、证据或风险发生变化。
- 哪些后续行动被触发。
- 哪些既有事件被推进、转化、终结或搁置。

### Step 2：过滤非事件

排除以下内容：

- 纯氛围。
- 纯设定说明。
- 纯心理延续。
- 未进入正文的后台状态。
- 只有风险分析但没有实际变化。
- 为了填满时间轴而制造的条目。

### Step 3：判断拆分粒度

按以下问题判断合并或拆分：

1. 输入是否不同。
2. 过程机制是否不同。
3. 输出是否不同。
4. 后续指向是否不同。
5. 主体是否不同。
6. 故事线归属是否不同。

输出不同或机制不同的候选，通常应拆开。只是连续动作且只形成一个输出的，通常合并。

### Step 4：输出事件候选表

建议使用轻量结构：

```yaml
eventCandidate:
  id: E000
  title: ''
  status: candidate
  chapterSpan:
    start: C0
    end: C0
  realTimeRange:
    start: ''
    end: ''
  primaryStoryline: ''
  referencedStorylines: []
  participants:
    characters: []
    locations: []
    organizations: []
    objects: []
  input:
    summary: ''
    previousEvents: []
  process:
    summary: ''
    conflictMechanism: ''
  output:
    summary: ''
    nextEvents: []
    stateChanges: []
  risk:
    wrongSplitRisk: ''
  sourceRef:
    files: []
    notes: ''
```

### Step 5：等待确认

用户确认前，不把候选写入运行投影 JSON。

确认可以是：

- 确认事件成立。
- 要求合并。
- 要求拆分。
- 改故事线归属。
- 降级为人物后台段。
- 暂缓进入本轮数据。

## 4. Codex 输出要求

Codex 输出事件采集结果时，必须说明：

- 为什么它是事件。
- 它改变了什么。
- 它和相邻候选如何区分。
- 它是否已确认。
- 它将进入哪个后续载体：源定义、采集表、事件脉络修订或暂存候选。

## 5. 禁止行为

- 不按段落机械拆分。
- 不按章节机械拆分。
- 不把故事线名直接当事件标题。
- 不把人物后台状态写成故事事件。
- 不为了页面视觉效果改变事件边界。
- 不在用户未确认时把争议候选写入正式投影数据集。
