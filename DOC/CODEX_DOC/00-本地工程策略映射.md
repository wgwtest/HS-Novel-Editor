# 本地工程策略映射

## 1. 文档根决定

本项目采用 `DOC/CODEX_DOC/` 作为本地 R&D 层正式文档根。它不替代根目录启动文档，也不替代已有历史原型包；它负责把项目从一次性验证工具整理为可长期推进的工程项目。

当前决定：

| 项 | 决定 |
| --- | --- |
| 正式文档根 | `DOC/CODEX_DOC/` |
| 快速启动入口 | `CODEX_START_HERE.md` |
| 历史设计说明 | 保留在 `设计说明/`，由正式索引引用 |
| 历史原型资产 | 保留在 `原型包/`，由正式索引引用 |
| 本地素材区 | `验证作品/`，只读引用，不进入 Git |

## 2. 文档治理依据

本地文档分类吸收以下标准控制族的职责，但不逐字复制其目录结构：

- `ISO/IEC/IEEE 12207`：软件生命周期过程。
- `ISO/IEC/IEEE 15289`：生命周期信息项与文档职责。
- `ISO/IEC/IEEE 16326`：项目管理计划。
- `ISO/IEC/IEEE 29148`：需求工程。
- `ISO/IEC/IEEE 29119`：测试文档与测试过程。
- `IEEE 1016`：软件设计描述。

本目录是项目内部工程治理层，不是对外交付包。若后续需要对外发布标准包，应在本目录之上建立投影层，而不是扭曲本地工程目录。

## 3. 本地扩展文档类

| 类别 | 本地目录 | 设置原因 |
| --- | --- | --- |
| 节点合同 | `05_节点合同/` | 让每个可执行节点有独立目标、边界、产物和验收方法。 |
| 测试文档 | `06_测试文档/` | 隔离验收入口、机测、人测和结论，避免散落在聊天和根目录。 |
| 过程文档 | `07_过程文档/` | 保存会话交接、历史计划和原型评审过程。 |
| 原型与附图索引 | `08_原型与附图/` | 记录原型包和截图事实源，不复制已有大体量资产。 |

## 4. 原型资产例外

Project Engineering Strategy 默认可将正式原型包放入 `08_原型与附图/`。本项目在采用策略前已经形成大量版本化 `原型包/`，并且当前页面、README 和启动命令均依赖这些路径。

因此本轮采用例外：

1. `原型包/` 保持为当前原型资产根。
2. `DOC/CODEX_DOC/08_原型与附图/` 先作为索引和治理入口。
3. 后续新原型是否迁入 `08_原型与附图/`，需要单独确认并同步启动文档。

## 5. Tracker 与远端协作

当前只确认 GitHub 仓库作为代码与文档远端：

```text
https://github.com/wgwtest/HS-Novel-Editor.git
```

尚未正式采用 GitHub Project 或其他远端 tracker。未确认前，不创建、不改名、不重构远端 tracker 字段或状态选项。WBS 和节点状态暂以本地文档为准。

## 6. 项目特殊约束

- `验证作品/` 中的正文、设定、事件脉络、成熟支撑材料只可本地读取，不可提交到仓库。
- 基准时间轴投影数据集不是最终结构化小说数据库。
- UI 页面不得绕过数据模型写死业务判断。
- 现有交接文档可能落后于源码，需要以“正式事实源索引 + 当前源码/校验结果”共同判断。
- 旧原型是设计过程证据，不自动成为当前实现标准。
