# 事件采集与人物参与策略规范 v1.0

## 1. 目标

本规范用于约束基准时间轴的数据采集和生成。它解决两个问题：

1. 故事线主事件不能只写“章节 + 故事线名”，必须有章内事件标题。
2. 人物子线不能复制主事件标题，必须描述该人物在该章节、该故事线中的参与形式。

本规范优先服务当前 `C1-C14 故事线分类预览` 基准时间轴投影数据集，后续可作为正文事件采集的执行准则。

## 2. 对象层级

### 2.1 故事事件

故事事件是全局时间轴主轨道上的事件条。

记录要求：

- 归属一条主故事线。
- 发生在一个章节或跨章节时间段内。
- 有独立标题，标题描述该故事线在该章节里发生的具体推进。
- 不能使用故事线名称本身作为标题。
- 可以关联多个参与人物。

故事事件回答的问题是：这条故事线在这一章推进了什么？

### 2.2 人物参与片段

人物参与片段是故事事件下方的人物子线条。

记录要求：

- 必须依附于一个故事事件。
- 必须有明确人物。
- 必须有 5-15 字的 `roleBrief`，推荐 6-12 字。
- `roleBrief` 描述该人物在该事件里的参与形式，而不是复述事件标题。
- 必须记录人物动作、关系变化和 Inspector 摘要。

人物参与片段回答的问题是：这个人物在这件事里具体承担了什么作用？

### 2.3 人物行 Inspector 聚合

人物行 Inspector 不是事件详情页，它是聚合视图。

当用户点击左侧某条人物行时，Inspector 应展示：

- 人物名。
- 所在故事线。
- 当前基准时间轴投影数据集中该人物在此故事线下的参与片段数量。
- 若干章内参与短述。
- 动作模式摘要。
- 关系变化摘要。
- 可进入事件详情的提示。

人物行 Inspector 回答的问题是：这个人物在这条故事线里整体怎么参与？

## 3. 字段规范

### 3.1 故事事件字段

运行数据中的故事事件至少包含：

| 字段 | 说明 |
| --- | --- |
| `id` | 稳定事件 ID |
| `kind` | 固定为 `story` |
| `track` | 主故事线 ID |
| `startAt` / `endAt` | 真实时间范围 |
| `title` | 章内事件标题 |
| `code` | 章节与故事线编码 |
| `visualState` | 视觉状态 |
| `input` | 本事件来源 |
| `output` | 本事件输出 |
| `risk` | 采集或叙事风险 |

### 3.2 人物参与片段字段

运行数据中的人物事件至少包含：

| 字段 | 说明 |
| --- | --- |
| `id` | 稳定人物参与片段 ID |
| `kind` | 固定为 `character` |
| `track` | 所属故事线 ID |
| `character` | 人物 ID |
| `participantName` | 人物显示名 |
| `startAt` / `endAt` | 与所依附故事事件一致 |
| `title` | 画布显示标题，必须等于 `roleBrief` |
| `roleBrief` | 5-15 字的人物参与短述，推荐 6-12 字 |
| `code` | 章节与人物编码 |
| `visualState` | 继承或调整自故事事件 |
| `action` | 该人物在事件中的具体动作 |
| `relationChange` | 该动作造成的关系变化 |
| `inspectorSummary` | Inspector 使用的摘要 |
| `input` | 可读动作描述 |
| `output` | 可读关系变化 |
| `risk` | 该人物参与的叙事风险 |
| `visibility` | `explicit` 或 `background` |

## 4. 采集规则

### 4.1 故事事件采集

从正文或章节梗概采集故事事件时，先问：

1. 这一章中哪条故事线被实际推进？
2. 推进动作是什么？
3. 这个推进是否改变了后续局面？
4. 如果标题去掉故事线名，是否仍能表达发生了什么？

不满足第 4 条的标题必须重写。

### 4.2 人物参与采集

对每个参与人物，必须单独采集：

1. 他/她在事件里做了什么。
2. 他/她是主动推动、被动承接、误读、压责、保护、转译还是诱导。
3. 他/她和其他人物或组织的关系有没有变化。
4. 这段参与是否正文可见。

人物参与不能只写“参与事件”。如果只能写“参与”，说明该人物不应生成子线条。

### 4.3 roleBrief 写法

`roleBrief` 必须：

- 控制在 5-15 字，推荐 6-12 字。
- 4 字以内只允许作为采集过程中的原始动作标签，不能直接进入画布显示。
- 优先使用“动作 + 对象/方向”的结构，例如“配置采访任务”“承接现场求助”“压住家族局面”。
- 避免“参与、相关、出现、推进”等空词。
- 避免复用主事件标题。

可接受示例：

- `吸收素材`
- `被迫承压`
- `误读授权`
- `建立归档`
- `截断羞辱`

不可接受示例：

- `参与电视台关系线`
- `C2 电视台关系线`
- `正式任务授权一周跟进`
- `在事件中出现`

## 5. 生成规则

当前 `C1-C14 故事线分类预览` 允许用规则生成候选人物参与字段，但必须满足：

- 生成结果写入基准时间轴投影数据集。
- `roleBrief` 必须稳定、可读、短，但不能短到只剩动作口号。
- 生成脚本不能在缺少故事事件标题时退回故事线名。
- 生成脚本不能让人物事件标题等于故事事件标题。
- 生成结果必须通过 `validate-character-participation.mjs`。

后续人工精修时，应优先修改源定义或采集文件，而不是直接手改基准时间轴投影数据集。

## 6. Inspector 消费规则

### 6.1 点击故事事件

Inspector 显示该故事事件的摘要、输入、输出和风险。

### 6.2 点击人物事件条

Inspector 显示该人物参与片段的 `action`、`relationChange` 和 `inspectorSummary`。

### 6.3 点击人物行

Inspector 聚合当前故事线下该人物的全部参与片段，显示：

- 参与片段数量。
- 章节与短述列表。
- 动作模式摘要。
- 关系变化摘要。
- 进入详情提示。

如果没有人物参与片段，则不应伪造内容，应明确显示“尚未采集人物参与片段”。

## 7. 验收规则

本规范对应的机器检查包括：

- `validate-character-participation.mjs`
- `validate-storyline-preview-quality.mjs`
- `validate-story-datasets.mjs`

验收标准：

- 人物事件标题不等于故事事件标题。
- 每个人物事件都有 `roleBrief/action/relationChange/inspectorSummary`。
- `roleBrief` 控制在 5-15 字。
- 人物行 Inspector 使用人物参与片段聚合，而不是模板文本。
- 浏览器中展开故事线后，人物子线显示短述。
